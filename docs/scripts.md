# Codename Eagle scripting

Almost everything dynamic in Codename Eagle - items, doors, vehicles, AI, lighting, mission logic, cutscenes - is driven by **scripts**. This is a guide to how that scripting system works: the execution model, the variables scripts see, the **callbacks** the engine fires on game events, and the **`REF` built-in functions** scripts call to act on the world.

> Most of the _facts_ here are reverse-engineered from the engine (`ce.exe`) and from mining all 869 shipped scripts (see [How these findings were recovered](#how-these-findings-were-recovered)). Where something is a best-effort reading rather than confirmed, it says so.

## The execution model

A script is **per-object**. Each placed mesh/object - the engine calls these **"projects"** - can have a script of the same name (`bird.scr` drives `bird`, `motobody.scr` drives the motorcycle body). One special script per level, `MAINSCR`/`mainscr`, drives the level itself (terrain, lighting, mission flow).

Scripts are authored in a small **C-like language** and compiled to **stack-machine bytecode** (`.scr`) by the `CPARSE` tool. The VM is **untyped and float-based**: every value on the stack is a 32-bit float, whether it represents a number, a flag, an object handle, or a string-table index. A call pushes its arguments, executes `call "Name"`, and leaves its result (a float) on the stack.

`cnetool` reads this format: `parseScript` decodes the bytecode, `decompileScript` recovers the C-like source shown throughout this doc, and `disassembleScript` gives a raw opcode listing.

### Anatomy of a script

A decompiled script is a flat list of **handlers** (functions) plus **module-level globals**:

```c
// GLOBAL/motobody.scr
float g0;                       // a persistent per-object global

Startup() {                     // engine callback: runs once when the object spawns
  REFSetProjectVars(MYSELF, 1, 0);
  REFSetItemTextureNr(MYSELF, 3);
}

EnterVehicle(player, vehicle) { // engine callback; known params get friendly names
  REFChangeVehicle(player, vehicle);
}

HitItem(object, type, player) { // engine callback: the object was hit
  if ((type == 2)) { REFSetVehicleStat(object, 1, 40); }
  REFRebirthItem(MYSELF);
}
```

The engine **fires handlers by name** in response to events (see [Callbacks](#callbacks-engine-fired-event-handlers)). Handlers can also call each other, and one object can invoke another's handler with `REFCallScript` / `REFCallScript_inst` (see [Calling other scripts](#calling-other-scripts)).

### Script resolution, instances & load order

Placements - the object instances a level spawns - come from the level's `World.dat`/`data1.bin` (see [formats.md](./formats.md)), and the instance names there carry an `_NN` suffix added by the editor (`flagbase_1`, `Tree1_48`).

**How an object finds its script** (binder `0x488040`, file index `0x4887f0`): when a placement loads, every `_` in its World.dat name is converted to `.` (`flagbase_1` → `flagbase.1`); the part before the first dot picks the project/mesh, and the **full dotted name is probed as a script file verbatim** before falling back to `<base>.scr`. So a compiled script named `flagbase.1` overrides `flagbase.scr` **for that one placement** - this is the per-instance script mechanism (the shipped CTF flag bases and Level8's switches use it). The file index merges the level dir and `GLOBAL/` (level wins for the same name), so full precedence is: `name.n` (level) > `name.n` (GLOBAL) > `name.scr` (level) > `name.scr` (GLOBAL). Per-instance files load as **separate script records** with their own globals.

Traps: the file index stores only the **first 12 characters** of each filename - a script whose filename exceeds 12 chars (base name > 8 with `.scr`) is silently invisible to the whole script system (`flagbase.scr` is exactly 12). And an instance extension starting with `s` is treated as a final name (never falls back to `.scr`). Bonus rule: after binding, a one-parameter `startup` receives `atof` of the text after the dot - so instances of a _shared_ `.scr` can learn their placement number.

**Two instance-numbering schemes** - don't conflate them:

- **Bind-order slots** (`0x488370`): all bindings of one script record are numbered by bind/spawn order, 0-based. Slots never shift on death; a respawn reuses the lowest freed slot of that script (and - decompile-level reading - a reused slot **keeps the previous instance's variable values**). This is what `REFGetProInstanceNr`, `REFInstanceNrExists`, and **`REFCallScript`**'s instance argument use.
- **Placement-name numbers**: **`REFCallScript_inst`** instead formats the _object name_ (`"%s.0%d"` then `"%s.%d"` for n < 10) and resolves it through the World.dat name registry - its `n` is the editor's `_NN` suffix. A miss is a **fatal error** (`AbortCall_inst`), unlike `REFCallScript`'s logged no-op. Duplicate placement names resolve to the last-loaded one. Quirk: `REFGetInstanceNr` probes only the zero-padded spelling for n < 10, so single-digit unpadded names (`flagbase_1`) are invisible to it, while `REFGetProjectNr` probes both.

**Level-load order** (from the `4711` entry `0x4438c0`; the restart path `0x47c6c0` is identical in shape): `scrcache.bin` preload (no handlers run) → **`mainscr` binds: its `Startup` fires** (no placed objects exist yet) → World.dat placements spawn in file order, **each object's `Startup` firing at its own spawn** → AI/routes load → player created: **`PlayerCreated`** → **`PrepareWorld`** → first frame. Scripter's contract: from `Startup` you may touch `mainscr` and anything placed earlier in the file; from `PlayerCreated` everything except later-joining players; everything is safe only from `PrepareWorld` on. Runtime spawns fire the new object's `Startup` synchronously inside the spawning call.

## Values & variables

There are only a handful of variable kinds a script deals with.

| Token                | Meaning                                                                                                                                                                                                                                                                                                                                         |
| -------------------- | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `MYSELF`             | A handle to **the object the script is attached to** (the "self" handle, the float `-1.0` internally). Passed as the object argument to most `REF` calls - `REFSetTTL(MYSELF, 0)` acts on this object. By far the most common token (≈4800 uses).                                                                                               |
| `gN` (`g0`, `g1`, …) | **Globals** - declared at module scope (`float g0;`) and **persistent for the life of the object instance**. Used as saved state (timers, flags, cached handles) and as **out-parameters**: many `REF` functions write their result into a global you pass by reference, e.g. `REFGetPosition(MYSELF, g2, g3, g4)` fills `g2/g3/g4` with x/y/z. |
| `vN` (`v0`, `v1`, …) | **Locals / parameters** - the arguments the engine (or a `REFCallScript` caller) passes into a handler. `HitItem(v0, v1, v2)` receives three; what each means depends on the callback.                                                                                                                                                          |
| literals             | A **number** (`40`, `-3.14`, `0.6`) or a **string** (`"vakt2.2"`, `"ANM\FLAG.ANM"`). Strings name projects, textures, sounds, dialogue keys, animation files, and AI instances.                                                                                                                                                                 |

There's also a parallel store of engine-side per-object state addressed **by index**, not by name: **`REFSetProjectVars(MYSELF, slot, value)`** (the single most-used function in the game, ~3000 calls). Each `slot` selects a specific object property - a flag bit, a numeric field like health, or a behaviour handler - decoded in [`REFSetProjectVars` slots](#refsetprojectvars-slots-reverse-engineered). `REFSetAIVars(name, slot, value)` is the AI equivalent - see [`REFSetAIVars` slots](#refsetaivars-slots-reverse-engineered).

## Callbacks (engine-fired event handlers)

The engine looks up handlers **by name** when game events occur. A script implements an event simply by defining a function with the matching name. The lookup is **case-insensitive** - scripts in the wild define `startup`, `StartUp`, and `startUp` interchangeably. Because these are known engine callbacks, the decompiler **normalizes them to the canonical casing** shown below (so every spelling of `startup` renders as `Startup`); author-defined handlers, which have no canonical form, are left as written.

This is the **complete** callback set: the engine looks every callback up by name through one function (`Script_FindHandler`, `0x487150`), so enumerating its callers in `ce.exe` 1.41 yields all of them. Each is then invoked by an **arity-specific dispatcher** that copies the engine's arguments into the handler's parameter slots: `CallScriptOne` (`0x487bf0`, 1 arg), `CallScriptTwo` (`0x487c40`, 2), `CallScriptThree` (`0x487cb0`, 3 - the `HitItem` fire site), or `CallScriptFour` (`0x487d30`, 4) - so the argument count and values are read directly off the dispatch site. Zero-arg callbacks go through the plain interpreter entry (`ExecuteScript`, `0x486d30`), which writes no parameter slots. (`AbortCall`/`AbortCall_inst` also go through the lookup but are the `REFCallScript`-not-found error path, not game callbacks.)

The parameter counts are the most common across all scripts that define each callback. Where a parameter's role is established - by mining how it flows into `REF*` calls across every shipped script, and reading the engine dispatch site - the decompiler also emits a **friendly name** instead of `vN` (so you'll see `SeePlayer(player)`, not `SeePlayer(v0)`); parameters whose role isn't established stay `vN`.

#### The use key

How one press of the use/enter key divides among the callbacks (input bit `0x2` in `player+0x60`, pumped per tick by `0x444bd0` → `0x4752f0`; all paths share the cooldown timestamp at `player+0x74`, ≈ 1 s = 36.4 ticks):

| Player state                                        | Fires                                             | On                                                                       |
| --------------------------------------------------- | ------------------------------------------------- | ------------------------------------------------------------------------ |
| On foot, enterable vehicle within 60 units          | `EnterVehicle(playerNr, vehNr)`                   | the nearest alive, unoccupied vehicle body's script                      |
| On foot (same press, always)                        | `Activate()`                                      | every scripted object with `dist − radius < 15` (no facing/LOS)          |
| On foot with an item held, press not consumed       | `UseItem(player, itemIdx)` + `ItemActivated(…)`   | facing (<~78°) `OBJECTCOLLISION` objects within `radius + 12` / the item |
| In a vehicle                                        | engine-side exit, then the `Activate()` broadcast | -                                                                        |
| (not a press) item lands in an empty character hand | `ItemUsed(1)`                                     | the item's script (spawn knife, `REFAddItem`, MP sync, savegame load)    |
| never                                               | `UseActive()`                                     | dead in every shipped engine                                             |

Arbitration: entering/activating stamps the cooldown to "now" (blocking item use that tick); item use stamps it to now + 36.4 ticks. The knife's melee attack reaches `UseItem`/`ItemActivated` through a separate fire site in the attack input path.

### Calling other scripts

Beyond engine callbacks, scripts define **their own handlers** and invoke them by name on a delay or on another instance - this is how levels build mission logic (`PlayerNoticed`, `Alarm`, `DropBomb`, `Open`, `KillBridge`, … are all author-defined, not engine events):

- **`REFCallScript(…, target, instance, "Handler", delay?)`** - call `Handler`. With a trailing number `delay >= 0` the call is **queued** to fire at `tick + floor(delay × 36.4)`, clamped to at least the next tick - so `0` = **next frame** (the SDK's `CALL_NEXT_FRAME`), never inline. With **no** trailing number the call executes **immediately, inline**, in the middle of the calling handler. `target` is a project name (or `""` for self), `instance` an instance number.
- **`REFCallScript_inst("target", instance, "Handler")`** - call `Handler` on a specific named instance, e.g. `REFCallScript_inst("PalDoor", 1, "Open")` (same delay rules).

> **Scheduling semantics** (delayed-call queue `0x46d420`, pump `0x46d500`):
>
> - The queue is pumped **once per world tick**, immediately after the tick counter increments - after physics and the engine→script callbacks, before timed object removals. Same-tick entries fire **FIFO** in schedule order.
> - **Delayed args are captured by value at schedule time** (variables are flattened to their current float). Inline (no-delay) calls instead copy variable args back to the caller on return - by-reference emulation exists only on the inline path.
> - **Recursion:** a handler re-invoking itself with delay `0` is safe (queued). Invoking itself with **no delay** recurses inline through native stack frames with no depth guard - a stack-overflow crash. The safe spelling is `0`; the dangerous one is _omitting_ the delay.
> - **Object death cancels pending calls.** Normal removal purges queued calls aimed at the object before firing `killed`; the unbind paths that skip the purge are caught at fire time (an unbound instance is skipped, never a use-after-free). Consequence: a `Killed()` handler that re-arms its own timer schedules into a dying binding - the chain ends there.
> - Missing target: if the _script_ exists but the instance doesn't, the call is skipped with a log line; a nonexistent script name, or a found script without the handler, is **fatal**.

> **Idiom - self-rescheduling periodic timer.** A handler that ends by scheduling _itself_ with a delay becomes a polling loop: `Foo()` whose last statement is `REFCallScript("", 0, "Foo", 0.5)` re-fires every 0.5 s for the life of the object. The loop is usually armed once from `Startup`/`PlayerCreated` (the first `REFCallScript("", 0, "Foo", t)`) and then keeps itself alive. This is how a script runs per-tick logic the engine has no dedicated callback for (e.g. periodically scanning every player slot via `REFGetPlayerInx`).

> **Idiom - the `-1` / `MYSELF` "none" sentinel.** `MYSELF` is the float `-1`, and several getters also return `-1` for "nothing": `REFGetPlayerInx` (empty slot), `REFPlayerHasVehicle` (on foot), `REFGetMother` (no parent), `REFGetPlayerNr` (no controlling player). Scripts exploit the coincidence - seed the out-var to `MYSELF` and branch on `!= MYSELF` - so the same constant doubles as "self" and "absent". `REFGetPosition` has a _different_ sentinel: a despawned object reads `y == -100000`.

> **Reading the decompiled call sites:** the decompiler reconstructs nested expressions from the stack VM using each `REF`'s arity. `REFCallScript` is **variadic** - its push order is `[handlerArg1, handlerArg2, …], target(str), instance(num), handler(str), [delay(num)]` - and is resolved **per call** the way the engine does (`FUN_00486bd0`: a number on the stack top is the `delay`, a string is the `handler`; the engine then pops `handler`/`instance`/`target` and passes the remaining deepest values as the target handler's declared parameters). The decompiler emits those leading handler arguments **positionally, before** the `target`/`instance`/`handler`/`delay` block - e.g. `REFCallScript(1, "RedLower", 0, "ChangeClothes", 0.5)` schedules `ChangeClothes` with one argument (`1`) and a `0.5`s delay. It recovers the arg count structurally (any operand pushed below the fixed block that is a bare value, not a leftover prior-call result, is an argument), so it neither drops real args nor mis-nests a preceding statement as a phantom first argument, and the source recompiles to identical bytecode. Early `return`s (mid-handler `ret`) are recovered too. Decompilation is still a best-effort reconstruction; cross-check with `disassembleScript` if a rendering looks off.

## The `REF` built-in functions

`REF` functions are the engine's **built-in script API** (the prefix is most likely **Refraction Games**, the developer). They are not script-defined; the `call` opcode names one, the engine resolves it through a registration table (name → C function pointer, near `0x4d2ed4` in `ce.exe`) and runs native code. There are **128** of them in 1.41 (`src/api/script.ts` → `REF_ARITY` holds the full name → arg-count list).

**Signatures below are reconciled against the engine implementation** (Ghidra pseudo-C of each `REF` impl) **and real call sites** mined from all 869 shipped scripts. Argument **types** (`str` / `num` / `obj`=object handle / `→out`=value written by reference) come from how each is popped and used; one representative example is given. Confidence:

- **Confirmed** - arg types and behaviour read from the engine implementation, usually corroborated by call sites.
- **Inferred** - behaviour only partly clear; some argument roles are a best reading. (No entry currently carries this: **all 128 are Confirmed**.)

Entries tagged **(unused)** are implemented in the engine but called by **no** shipped script (verified by a raw search of all 2764 `.scr` across every version). A few turn out to be **no-op or disabled stubs** - flagged inline.

> Names are matched case-insensitively, so the mined data contains spellings like `REFCAllScript`, `REFPlayDLG`, `REFShowCutScene` - all the same functions as their canonical-cased entries below.

#### The team model

Two **independent** per-character fields (players and AI share one 0x330-byte character struct, reached from a script object via `proj+0x2ec`):

- **Faction bitmask, `char+0x28`** - what `REFSetTeam` and AI-var `TEAM` write (the AI-var write also propagates to grouped members). Consumed **only by AI target selection** (`0x425f00`/`0x4263d0`): a candidate sharing any bit is a teammate (AND test; whole-value equality is used as a fast same-team skip). It does **not** gate damage. Defaults: players `0` (zeroed struct), every AI `2`. Script values: `1` = the player's side, `2` = the enemy, `3` (=1\|2) = friend of **both** sides - this is level1's uniform-disguise mechanic (`ChangeClothes` → `REFSetTeam(3)`) and the safe-civilian setting - and `4` = a bit shared with nobody (panicked civilians: targetable by all, loyal to none).
- **MP team byte, `player+0x322` bit0** - what `REFIsTeamA` reads: `0` = red, `1` = blue. Assigned at join (`0x472c30`; request `2` = auto-balance onto the smaller team, 8-per-team cap; forced `1` in SP/deathmatch), preserved across respawns, network-synced. In team modes **this bit** - not the faction mask - gates friendly fire (`0x416370`), team chat, team score totals, team skins, name-tag colors, and team spawn-point selection. CTF: `flaga` can only be taken by red and `flagb` by blue (each side carries the other's flag). The game-mode global `0x4de794` is `0` deathmatch / `1` ctf / `2` teamplay.

The only point where the two fields meet is the MP change-team command (`0x479aa0`), which toggles both and forces a respawn. MP has no AI, so the faction mask is effectively inert there.

## `REFSetProjectVars` slots (reverse-engineered)

`REFSetProjectVars(obj, slot, value)` is **not** a generic key/value store - its implementation (`0x48af20`) is a `switch (slot)` with a **22-entry jump table** (`0x48b5c0`, slots `0`-`21`; out-of-range is a no-op). Each slot writes a _specific_ field on the engine's project object or toggles a _specific_ bit in its property word `[proj+0x2a8]`. Boolean slots test `value != 0` (`set bit` if true, `clear bit` if false); numeric slots store the value directly. Most flag changes then call `0x4666a0` to recompute derived object state.

The **field offsets and operations below are confirmed from the disassembly**; the _gameplay meaning_ of each bit/field is decoded by combining three signals - the engine read sites, the recompute routine, and which object _types_ set each bit.

`[proj+0x2a8]` is the object's **high-level property bitfield**. The renderer gates drawing on bit `0x2`; the **recompute routine `0x4666a0`** (called after most flag writes) packs the property bits into the derived **collision-class byte `[proj+0x2b0]`** (raw-asm-verified truth table; `props` = `+0x2a8`):

| `0x2b0` bit | expression                                                   | meaning                                                                  |
| ----------- | ------------------------------------------------------------ | ------------------------------------------------------------------------ |
| `0x01`      | `[+0x2d0] != -1`                                             | dying/despawning (corpse timer armed)                                    |
| `0x02`      | `!(props & 0x4 && props & 0x40)`                             | NOT a free dynamic body (not MOVE+LANDCOLLISION together) - most scenery |
| `0x04`      | `!(props & 0x20)`                                            | not touchable (no OBJECTCOLLISION)                                       |
| `0x08`      | `(props & 0xc) != 0`                                         | movable (MOVE or ROTATE)                                                 |
| `0x10`      | inactive, or `(props & 0x60) == 0`, or no collision geometry | fully collision-exempt - never enters the broadphase grid                |
| `0x20`      | `[+0x2b1] & 1`                                               | is-projectile                                                            |

The **pairwise filter** (in the pair-builder `0x466750`) rejects a pair when: the two bytes share any of bits `0x07` (two dying, two non-dynamic - why static scenery never collides pairwise - or two untouchables); one side is untouchable and the other movable (either way); both are projectiles (`0x20`); both have `IMMATERIAL` (read directly from `0x2a8` - items pass through each other); or both belong to the **same controllable** (`+0x2ec` equal - a projectile never hits its shooter, vehicle parts never self-collide).

> **Contact-handler convention:** per-frame behaviour/damage handlers installed at `[proj+0x30c]` (slots 10/15/20 below, vehicle damage `0x436e10`, character damage `0x408bd0`) are called on first contact of a pair as `handler(self, other, impactSpeed, contact, contact2)` and their byte return is cached at `+0x2b3`:
> **return 1 = solid (physics response proceeds), 0 = pass through**.

Confidence: **Confirmed** = verified at an engine read site; **Likely** = strong inference from the engine grouping + a clean object-type cluster; **Guess** = weak.

Every name has been cross-checked against the `0x48af20` switch; six with non-obvious meanings (`MASS`, `AFFECTEDBYEXPLOSION`, `NO_ZBUFFER`, `SOLID_MATERIAL`, `IMMORTAL`, and the `MOVE`/`ROTATE` pair) were each pinned down at their engine read sites - see the per-slot notes.

#### `WEAPON_TYPE` ids

Names are the developers' own, decoded verbatim from `data4.bin` (SP) / `mdata4.bin` (MP). #### Infantry fire rate, weapon switching, and the "8 trick"

Fire rate is enforced in **one place**: the input applier `0x47ab50` (runs per player per tick, inside every lockstep peer's simulation) ends in the gate `0x47ab20`, which **eats the fire-button bit** while `now - [chr+0x68] < [chr+0x6c]`, and otherwise stamps `[chr+0x68] = now`. The two fields on the character controller:

- `[chr+0x68]` - fire-cycle start tick (the last shot).
- `[chr+0x6c]` - fire-cycle length in ticks: the fire-time stats applier `0x44e6b0` copies the weapon row's FireDelay field (tick-scaled by the loader `0x44dee0`) into the firing controllable, and the per-tick character update refreshes it from the held weapon. MP values (`mdata4.bin`): bazooka / snipergun / grenade / gasw 2.0 s, rifle 1.0 s, kpist 0.125 s.

Weapon _switching_ is a separate state machine on the controller:

- `[chr+0x2c4]` - weapon-ready state: `0` ready, `3` alt-ready (sniper zoom, detonator armed), `4` taking up a weapon, `5` → `3`, `1`/`2` transient putting-away states.
- `[chr+0x2c8]` / `[chr+0x2cc]` - select timestamp + take-up duration. `SelectWeapon` (`0x4121f0`; reached from the input-command nibble `(chr+0x60 >> 3) & 0xf`: `1` = cycle `0x415c30`, `2`-`0xb` = `SelectWeapon(n-1)`) always enters state 4 with a **hardcoded 36.4-tick take-up** - including when re-selecting the weapon already held.

**The bug** (the community's "8 trick"): while a switch is in progress, the character update (the giant per-tick function `0x412810`) **zeroes `[chr+0x68]`** (the 7-byte store `c7 46 68 00 00 00 00` at `0x414be8`) every tick **except the completion tick**, whose gate commit survives - so the take-up replaces the remaining fire cooldown, and the first trigger pull after any switch fires. Any weapon whose FireDelay exceeds the take-up (bazooka, snipergun, grenade, gasw at 2 s) can therefore fire noticeably faster by re-selecting it after every shot (bazooka = key 8, usually macroed to a mouse button). Switching _away and back_ abuses the same store, so a fix inside `SelectWeapon` (ignore same-weapon selects) is insufficient. Full analysis and a fix discussion: [game-flow.md](./game-flow.md#the-fire-pipeline-weapon-cooldowns--the-8-trick).

`REFSetAIVars(name, slot, value)` is the AI-side analogue. The wrapper (`0x48c4e0`) resolves the target - `name == ""` means the calling object, otherwise the named AI instance - and forwards `(name, obj, slot, value)` to the worker **`0x4256d0`**, a `switch (slot)` with a **12-entry jump table** (`0x42585c`, slots `0`-`11`). | 10  | `CLIPGAS`     | Magazine gas (max `MAX_PLAYER_GAS` = 100).                                                                                                                                                 |

`REFReadPlayer` reads the same table but is a **separate stat table from the vehicle one** (`REFReadVehicleStat`), so indices don't line up between them.

**`REFSetScore(player, mode, delta)`**:

- Arg 0 is a **player handle** (resolved: the impl bounds-checks it against the player count and indexes the player table). The SDK header's `TEAMA` = 0 / `TEAMB` = 1 names are **wrong** for this function - shipped scripts passing a player handle (`flaga.scr`: `REFSetScore(player, FLAG_SCORE, 5)`) have it right. Team totals are maintained automatically from the player's MP team bit.
- `mode`: `FLAG_SCORE` = 0 (score column), `KILLED_SCORE` = 1 (kills column); `delta` adds.

**`REFSetViewMode(mode, elev, azim, zoom)`** - `mode`: `INSIDE` 0, `OUTSIDE` 1, `FRONTFLYBY` 2, `BEHINDCHASE` 3. (The SDK calls the angle args `ax/ay/az`; they are elevation, azimuth, and zoom - see the builtin's entry for per-mode use. Two more modes exist on keyboard keys only: front-chase `0x02` (F3) and a fixed drop-cam `0x08` (F4) - not reachable from scripts.)

**`HitItem(otherInx, playerType)` callback** - `playerType`: `LOCALPLAYER` 0, `OTHERPLAYER` 1, `VEHICLE` 2. (Confirmed at the fire site; the `player` arg is an uninitialized register when `VEHICLE`.)

**`REFGetProVect(obj, type, →out)`** - `type`: `DOF` 0, `UP` 1, `RIGHT` 2. Axis component vars: `XVAR` 0, `YVAR` 1, `ZVAR` 2.

**`REFSetWeatherType(type)`** - `OFF` 0, `SNOW` 1, `RAIN` 2.

**`SetLight` type** - `SPOTLIGHT` 0 (view-gated searchlight glare), `NORMALLIGHT` 1 (plain omni). Modes 2 (off) and 3 (dynamic, re-callable per tick) exist engine-side without SDK names - see the `REFSetLight` entry.

**General literals** - `MYSELF` = -1; `THIS` / `TOPLAYER` / `CAMERA` / `CALL_NEXT_FRAME` = 0; `TRUE`/`ON` = 1, `FALSE`/`OFF` = 0; `LASTFRAME` = 65535 (`0xFFFF`, for `REFAddAnimation`/`REFLoadAnimation`). Player capacities: `MAX_PLAYER_BULLETS` 200, `MAX_PLAYER_SHELLS` 50, `MAX_PLAYER_GAS` 100.

## Multiplayer determinism

CE multiplayer is **deterministic input-lockstep** ([`network.md`](./network.md) §5), and scripts run _inside_ the simulation on every peer - so script-visible randomness has to agree across machines. It does, by construction:

- The only `srand` in the engine is `srand(4711)` (yes, the autostart magic number), run once per level load to fill a fixed **512-entry random table** - identical on every machine, never time-seeded.
- Gameplay reads the table through the **sim cursor**; particles/sounds/environment read through a separate **FX cursor** - heavier rendering on one machine cannot shift the gameplay stream.
- The sim cursor resets to 0 at MP game start and at **every player spawn/respawn** (lockstep events executed at a wire-specified tick), a periodic hard re-sync; one sim draw is burned per tick to keep the stream moving.
- Divergence is **detected, not corrected**: a player-position checksum every 36 ticks; a mismatch is "WORLD OUT OF SYNC" in `netlog.txt` and a kick.

Script-facing rules:

| Builtin                      | MP verdict                                                                                                   |
| ---------------------------- | ------------------------------------------------------------------------------------------------------------ |
| `REFRandom`, `REFRandomItem` | **Lockstep-safe** - they draw from the synced sim stream (no MP gating exists or is needed).                 |
| `REFGetTime`                 | **Not safe** - wall-clock seconds, peers disagree. Fine for local/cosmetic use; never branch gameplay on it. |

## VM limits (engine-side)

Practical ceilings read from the loader/interpreter (`0x488b00`/`0x486d30`):

- **Handler names: 31 chars** (32-byte field in the 0x28-byte handler record). The old "7-char cap" was `CPARSE`-side only.
- **Script base names: 27 chars** - `BindScript` builds `name + ".scr"` in a 32-byte stack buffer with **no bounds check**; 28+ smashes the stack.
- **String operands (literals, `REF` names): 255 bytes** - loaded through an unchecked 256-byte buffer.
- **Globals: ≤ 16383 per script** (u16 byte count ÷ 4), and a hard **65536 live var slots across all bound instances** (u16 bases; beyond that, silent wraparound).
- The value stack and all scheduler pools grow unbounded in 64-node blocks that are never returned; script instances are never freed (recycled by same-def rebinding).
- Inline by-variable args use a 256-entry save stack (`"PushVarnr stackoverflow"` when exceeded).

## Calling convention (how arguments are passed)

Reverse-engineered from the implementations. A `REF` function reads its arguments off the script stack with type-checking pop helpers:

- **`0x486c30`** - `PopStackValue` as a **number** (asserts _"PopStackValue() found a string on stack"_ otherwise). The popped float is then either `fstp`'d to a float destination (a real **float** arg) or passed to **`0x494130`** (an `__ftol` float→int helper) and used as an **int / handle / enum / index**.
- **`0x486d10`** - `PopStackValue` as a **string** (asserts _"found a non string"_).

> **Ghidra trap:** the pseudo-C routinely swallows FPU instructions sitting between a pop and the `0x494130` conversion into an opaque `__ftol()` call - a `fadd` (so an **add** reads as a plain store: `REFChangePlayer`), a `fmul` (so a **seconds→ticks conversion** reads as a raw int: `REFLockCameraOnObj`), or a `fstp st(0)` (so a **discarded arg** looks consumed: `REFPlayDlg` arg 3). Any arithmetic-sensitive reading must be cross-checked against the raw x86 at the implementation address.

Caveat for anyone automating this: **string arguments aren't reliably distinguishable at the pop site.** Some functions take strings as real string pops (`REFSetPlanet`), but others pass them as a numeric **string-table index** popped as a number (`REFSetLandscape`'s `land`/`sky`). Functions also pop a few trailing _context_ values (caller object/instance) beyond their declared args. Mining the **call sites** (above) sidesteps this: a `"literal"` at a position is unambiguously a string there.

**Nothing checks arity at compile time.** `CPARSE` (the `.scr` compiler) is a generic ParGen/yacc C front-end - it contains **no `REF` names or prototypes**, so a call `name(a, b, c)` simply emits _n_ argument pushes followed by `call "name"`, whatever _n_ is. Its only diagnostics are syntax/lexical errors. Each `REF` implementation pops a **fixed** number of values, so arity is enforced only at runtime: supply too few and the extra pops underflow to defaults; too many and the surplus stays on the stack as residue (the leading-nested-call artifact noted above). This is also why a misspelled/miscased name like `REFCAllScript` still compiles and runs - neither the compiler nor the engine's case-insensitive lookup rejects it.

## How these findings were recovered

Two evidence streams feed this doc, and anyone wanting to verify or extend it can reproduce both:

**Engine side.** The impl addresses, slot switches, the script dispatch family, and the collision packing come from reading a **headless Ghidra decompilation of `ce.exe` 1.41**, exported as one greppable pseudo-C file (grep a hex address like `48af20` or a resolved name to jump to a function). Address conventions: `ce.exe` loads at image base **`0x400000`**, so `file_offset = vaddr − 0x400000`; the menu/server-browser logic lives in **`MENUDLL.DLL`**, a separate binary decompiled the same way, and as a DLL its image base is **`0x10000000`** (`file_offset = vaddr − 0x10000000`) - see [`network.md`](./network.md) §2. Two caveats for anyone repeating this:

- **Ghidra's auto-analyzer misses functions** reached only through data pointers or hidden behind interleaved switch tables - in `ce.exe` that includes the character death callback `0x408bd0`, the wreck/crush handler `0x432520`, the `PatrolEnd` fire site `0x429050`, the per-tick character input/weapon update `0x412810`-`0x415a68` (one giant function), and the use-key pump `0x444bd0`. **Force function creation at those addresses before exporting** or they won't appear in the dump at all. Corollary: a grep of a dump cannot prove absence - cross-check with a raw-binary scan (rel32 call operands, immediates) or the Ghidra project's real xref database, which also resolves indirect calls and data references that text search can't.
- The output is **pseudo-C, not compilable C** (synthetic types `undefined4`/`float10`, `__fastcall`, undeclared `DAT_*` globals, `CONCAT`/`SUB` intrinsics) - and it hides FPU arithmetic inside `__ftol()` (see the [Ghidra trap](#calling-convention-how-arguments-are-passed) note), so arithmetic-sensitive readings need the raw x86.

**Script side.** The signatures, examples, and per-function caller lists come from decompiling **all 869 shipped `.scr` scripts** (2764 across every game version) with this library's `decompileScript` API and mining the call sites of every `REF` builtin across the output: a `"literal"` at an argument position is unambiguously a string there, which sidesteps the pop-site ambiguity described in [Calling convention](#calling-convention-how-arguments-are-passed), and clustering which object scripts set which slot/flag values is what decoded most of the slot tables. `parseScript`, `disassembleScript`, and `compileScript`/`compileSource` cover the rest of the round trip when a decompiled rendering needs cross-checking at the opcode level.
